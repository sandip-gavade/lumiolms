  2. Two ways to authenticate

  Option A — Swagger UI (easiest for exploring)

  1. Open http://localhost:3001/docs
  2. Since you have no tenant yet, create one first via curl (below) — Swagger UI can't set the x-tenant-id header globally, so signup/login is easier from the terminal the first time.
  3. Once you have an accessToken, click Authorize (top right of /docs) and paste it in as Bearer <token>. Every subsequent request in the UI will carry it.
  4. Catch: routes still need tenant resolution. Since you're hitting localhost (no subdomain) and Swagger UI won't add x-tenant-id for you, tenant-scoped GET routes without a token will fail. Once you're using a

  The one gotcha: tenant resolution

  Every request needs a tenant. Two ways it gets resolved (see TenantContextMiddleware):

  ┌──────────────────────────────────────┬───────────────────────────────────────────────────────────────────────────────┐
  │              Situation               │                            How tenant is resolved                             │
  ├──────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ You have a valid Bearer token        │ From the JWT's tenantId claim — automatic, no header needed                   │
  ├──────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ No token yet (login/signup/browsing) │ From x-tenant-id: <tenant-uuid> header (dev only — production uses subdomain) │
  └──────────────────────────────────────┴───────────────────────────────────────────────────────────────────────────────┘

  So: pass x-tenant-id on unauthenticated calls (login, signup, public catalog browsing); once you have a token, Authorization: Bearer <token> alone is enough.
