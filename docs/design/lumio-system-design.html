Architecture, modules, and local project layout

A multi-tenant learning platform: organizations get a branded instance, students learn, instructors author, and admins run their tenant. This page maps the moving parts to how the codebase is actually organized.

01

Architecture

Every request enters through a tenant's subdomain, resolves to that org's context, and is served by a single stateless API fleet — no per-tenant infrastructure. Async work (email, certificates, webhooks) is offloaded to a worker fleet sharing the same Redis instance, so request latency never waits on a PDF render or an outbound email.

02

Backend modules

NestJS's module system mirrors what a modular monolith needs here — each domain owns its controllers, services, and DTOs, with tenant scoping enforced centrally rather than per-query. tenant-scoped modules read the org from the request context; platform modules cut across all tenants.

03

Frontend structure

Next.js App Router with route groups that map directly to the audience: a public marketing/catalog surface, the authenticated learner app, an instructor console, and an org-admin console — each behind its own layout and role guard, sharing one MUI theme.

(marketing)

Landing, course catalog, course detail — public, tenant-branded via subdomain.

(learner)

My Learning, lecture player, notes, Q&A, profile & settings.

(instructor)

Course authoring, curriculum builder, revenue & ratings dashboard.

(org-admin)

User & course moderation, branding, tenant analytics, billing.

04
